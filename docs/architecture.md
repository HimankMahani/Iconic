# Iconic Architecture

A tour of how the code is organized, written for **new human
contributors** who want to understand the codebase before sending a PR.
If you're an AI coding agent, see [`dev/AGENTS.md`](dev/AGENTS.md) for
the AI-specific framing.

---

## What Iconic does (one sentence)

Scans a folder tree, maps each folder's name to an SF Symbol (or emoji),
renders a multi-resolution custom icon, and writes it back to the file
system via `NSWorkspace.setIcon(_:forFile:options:)`.

## Layered architecture

```
┌────────────────────────────────────────────────────┐
│ UI layer (SwiftUI)                                 │
│   IconicApp → ContentView / PreferencesView / ...  │
│                                                    │
│   reads via @EnvironmentObject, writes via closures│
└────────────────────────┬───────────────────────────┘
                         │
┌────────────────────────▼───────────────────────────┐
│ Coordinator (IconicViewModel)                      │
│   Owns: items, selection, undo/redo, toasts,       │
│         matching mode, search/filter, dry-run,     │
│         background monitor, custom mappings, ...   │
│   All @MainActor; heavy work → Task.detached       │
└────┬────────┬────────┬─────────┬─────────┬─────────┘
     │        │        │         │         │
     ▼        ▼        ▼         ▼         ▼
  Scanner  Matcher  Renderer  Applier  Stores / Services
```

## Data flow (single scan)

```
User picks a folder (or drops one)
   │
   ▼
IconicViewModel.adoptRoot(url)
   │
   ▼
IconicViewModel.scan(url)           ← async
   │
   ├─► FolderScanner.scan(url)      ← Task.detached, off-main
   │      → [URL]                   (subfolder URLs)
   │
   ├─► For each URL (in parallel via withTaskGroup):
   │      │
   │      ├─► Matching (priority order):
   │      │    1. RulesStore.firstMatch(for:)
   │      │    2. SmartContentDetectionStore + FolderTypeDetector
   │      │    3. CustomMappingsStore (user overrides)
   │      │    4. GeminiService.matchFolders(...)   (if AI enabled + key present)
   │      │    5. SymbolSearchEngine.search(folderName:)
   │      │    6. SymbolMapper.symbolWithConfidence(...)
   │      │    7. Fallback: SymbolMapper.fallbackSymbol
   │      │
   │      ├─► ColorPalette.assignColor(for:)         (if autoColor enabled)
   │      │
   │      └─► IconRenderer.makeIcon(...)              ← Task.detached
   │             → NSImage (multi-rep)
   │
   ▼
User reviews (search, filter, dry-run)
   │
   ▼
IconicViewModel.apply(item) or applyAll()
   │
   ▼
IconApplier.apply(item, to: url)
   │
   ▼
NSWorkspace.shared.setIcon(image, forFile: url.path)
```

The `FolderItem` row carries the rendered preview, all the visual
settings, the match source, and the lifecycle status (`.pending`,
`.applying`, `.applied`, `.restored`, `.failed(message)`).

## State management rules

- All UI-touched state is `@MainActor`.
- All non-trivial work (scanning, rendering, applying, Gemini calls)
  is `Task.detached` or `Task { … }` from `@MainActor` types.
- Stores are `@MainActor final class … : ObservableObject`. Publish
  with `@Published`. Persist to `UserDefaults` on every mutation.
- Undo/redo lives in a single `IconicUndoManager` instance on the
  view-model; both Apply All and Restore All record the per-folder
  previous state so a single undo reverses the whole batch.
- Concurrency: `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` is set in
  the Xcode project, so the entire default isolation is main-actor
  unless explicitly opted out. `SWIFT_APPROACHABLE_CONCURRENCY = YES`
  and `SWIFT_UPCOMING_FEATURE_MEMBER_IMPORT_VISIBILITY = YES` are
  enabled for forward compatibility with Swift 6.

## Where to add things

| You want to… | Edit | Where in the file |
|---|---|---|
| Add a new keyword → symbol mapping | `Iconic/SymbolMapper.swift` | Append to `builtInMappings` (line ~18) |
| Add a new emoji mapping | `Iconic/EmojiMapper.swift` | Append to `builtInMappings` (line ~18) |
| Add a new themed color palette | `Iconic/ColorPalette.swift` | Append to `palettes` dict + extend `categoryKeywords` (line ~80) |
| Add a new folder type detector (e.g. "Rust project") | `Iconic/FolderTypeDetector.swift` | Add a `hasFileWithExtension` / marker check; add a case to the switch in `detectType(at:)` |
| Add a new settings tab | `Iconic/PreferencesView.swift` | New `@ViewBuilder` + add to the `TabView` |
| Add a new user-facing row in the main list | `Iconic/FolderRowView.swift` | Follow the existing closure-callback model; do **not** store state inside the view |
| Add a new persistent preference | `Iconic/PreferencesStore.swift` | Add a new `enum XxxStore { static var foo: Bool }` backed by `UserDefaults` |
| Add a new Gemini prompt variant | `Iconic/GeminiService.swift` | Extend `buildPrompt(...)` or `buildEmojiPrompt(...)` |
| Add a new file type that triggers content analysis | `Iconic/FolderContentAnalyzer.swift` | Add a marker field to `ContentAnalysis` and a check in `analyze(_:)` |

## Build & test

```sh
# Build
xcodebuild -project Iconic.xcodeproj -scheme Iconic -destination 'platform=macOS' build

# Run all tests
xcodebuild -project Iconic.xcodeproj -scheme Iconic -destination 'platform=macOS' test

# Regenerate the SF Symbol / emoji metadata (rare)
swift Scripts/extract_symbols.swift    # writes Iconic/Generated/SymbolMetadata.swift
swift Scripts/extract_emoji.swift      # writes Iconic/Generated/EmojiMetadata.swift
```

The first cold build with the regenerated metadata takes ~30 minutes
because `Iconic/Generated/SymbolMetadata.swift` is 711 KB of case data
covering 7,968 SF Symbols. Subsequent builds are fast.

## What you should NOT edit

- Anything under `Iconic/Generated/`. These are derived from Apple
  system plists; hand-edits will be overwritten next regeneration.
  See [`THIRD_PARTY_NOTICES.md`](THIRD_PARTY_NOTICES.md) for provenance.
- The `Iconic/Iconic.entitlements` and `Iconic/PrivacyInfo.xcprivacy`
  files are part of Apple's notarization requirements. Changes here
  should be discussed in an issue first.
- `Iconic.xcodeproj/project.pbxproj` uses Xcode 16's
  `PBXFileSystemSynchronizedRootGroup`, which means files added to
  the `Iconic/` directory are **automatically** included in the build.
  You usually do not need to edit the pbxproj to add a source file.
